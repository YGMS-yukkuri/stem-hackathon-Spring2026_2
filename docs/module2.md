# モジュール2: モジュール1詳細実装方法についてのドキュメント

このドキュメントは、`モジュール1` で定義した要件を基に、具体的な実装方法や設計のガイドラインを提供するためのドキュメントです。以下の内容を参考にして、モジュール1の要件を満たすように実装を進めてください。

---

## 主な機能
- ユーザーの現在位置の取得して、近隣の施設やサービスを検索する機能
- 施設やサービスの情報を表示する機能
- 施設やサービスの情報を保存・編集・削除する機能
- 施設やサービスの情報を共有する機能(できれば)
- 施設やサービスの情報を検索・フィルタリング・ソートする機能
- 選択した候補からランダムに選択する機能
- 保存したものからランダムに選択する機能
- 目的地までの距離や時間の表示機能

## 目的
- ユーザーの現在位置から近隣の施設やサービスを検索し、表示・保存・共有・ランダム選択を行えるようにする。
- 寄り道を楽しむために必要な施設や店舗を検索する。
- ユーザーが保存した施設やサービスの情報を管理できるようにする。
- ユーザーが選択した施設やサービスの情報をもとに、目的地までの距離や時間を表示することで、寄り道の計画を立てやすくする。

## 概要
- 入力: ユーザーの現在位置（緯度/経度）および検索条件（種別、距離、キーワードなど）
- 出力: 施設一覧（名称、住所、営業時間、電話番号、URL、緯度/経度、推定所要時間/距離）

## 要件（モジュール1の要件を具体化）
- 要件1: ユーザーが現在位置をシステムに提供できること。
	- 実装例: ブラウザの Geolocation API または手動入力で緯度/経度を受け取る。
- 要件2: システムが現在位置をもとに近隣の施設やサービスを提供できること。
	- 実装例: 外部の無料API（例: OpenStreetMap / Nominatim / Overpass API）を利用して検索結果を取得。
- 要件3〜7: 施設情報の表示、複数候補表示、ランダム選択
	- UI: 結果リスト + 詳細モーダル
	- ランダム選択: ユーザーがチェックした候補群からランダムに1つ選ぶ機能
- 要件8〜16: 保存・閲覧・編集・削除・検索・フィルター・ソート・保存からのランダム選択
	- 永続化: ローカルストレージ（プロトタイプ）/ サーバーDB（実運用）
- 要件17: バックエンドとフロントエンドの連携
	- API仕様を以下に定義
- 要件18: 外部API利用のドキュメント提供（無料API優先）

## 利用者フロー（ユーザー視点）
1. 位置共有の許可ダイアログ表示 → 許可されれば自動で位置取得
2. 検索条件を入力して検索実行
3. 施設リスト表示（ソート/フィルタ可能）
4. 施設を選択 → 詳細表示（ルート表示ボタンあり）
5. 保存ボタンでお気に入りへ追加（サーバー or ローカル）
6. 複数選択してランダム選択ボタンを押すと1件が選ばれる
7. 保存した施設の一覧から編集・削除・ランダム選択が可能

## データモデル（例）
- Facility
	- id
	- name
	- address
	- lat, lng
	- phone
	- url
	- opening_hours
	- source (外部APIの識別子)
- SavedItem
	- id
	- userId
	- facilityId
	- note
	- createdAt

## UI要件・ワイヤーフレーム
- 検索画面: 位置表示、キーワード入力、半径スライダー、検索ボタン
- 結果一覧: 名称、簡易住所、所要時間/距離、保存ボタン、詳細リンク
- 詳細モーダル: フル情報、ルート表示ボタン、共有ボタン
- 保存一覧: 保存したアイテムの編集・削除、複数選択チェックボックス、ランダム選択

## 施設検索やルート表示（外部マップ連携）
- 実装案: Leaflet + OpenStreetMap を利用
- 代替案（無料）: OSRM や GraphHopper の無料プラン/自己ホストを検討

## エラーハンドリング
- 位置取得エラー: ユーザーへフレンドリな説明と手動入力を促す
- 外部APIエラー: フェールオーバー（別API）またはエラーメッセージ表示
- 保存失敗: 操作取り消し・再試行ボタンを提供

## テストケース（例）
- ユニット: 検索パラメータから正しく外部APIクエリを組み立てられる
- 統合: `GET /api/search` が外部APIを呼び結果を正しく整形して返す
- E2E: 位置許可から検索、保存、保存一覧表示、削除、ランダム選択までの一連動作

## 依存関係
- フロント: Leaflet, axios など
- バックエンド: express (Node), requests (Python) 等のHTTPクライアント
- 外部API: OpenStreetMap / Nominatim / Overpass / OSRM など（無料優先）

## 実装ノート
- プライバシー: ユーザー位置は必要最低限のみ送信し、保存する場合は明示的同意を得る
- レート制限: 外部APIの制限を考慮してキャッシュとクエリ頻度を制御する
- ローカル開発: 外部APIモックを用意するとテストしやすい

## 確定仕様（2026-03-16 ユーザー回答反映）
- 技術スタック
	- フロントエンド: React を採用する。
	- バックエンド: Windows 11 開発環境での扱いやすさを考慮し、Node.js（Express）を採用する。
- 保存方式
	- 主要な保存先はブラウザ内ストレージ（LocalStorage 優先、必要に応じて Cookie 併用）とする。
	- DBは初期実装では使用しない。
	- 将来的な拡張として、Node.js 側でファイルベース保存（例: JSONファイル / lowdb など）を選択可能とする。
- 認証・識別
	- ログイン認証は実装しない。
	- クライアント初回利用時に UUID を自動生成し、LocalStorage または Cookie に保存して userId として扱う。
- 外部API優先順位
	- 施設検索は Nominatim / Overpass API を優先利用する。
	- ルート情報が必要な場合のみ、補助的に OSRM 等を利用する。
- 共有機能
	- 共有機能はMVPスコープから除外可能（未実装可）とする。
